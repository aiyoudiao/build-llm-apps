## Day 5-2 总结

主要是在「RAG 系统进阶优化」上，从检索重排序 (Rerank)、召回率提升 (Multi-Query)、复杂 Query 改写与意图识别，到知识库的深度加工与维护。大概可以分成四条线：Rerank 重排序、召回率优化、Query 改写策略、以及知识库工程化。

### 记忆卡片
- **Rerank 重排序**：在检索回来的文档候选集中，使用更精准的 Rerank 模型（如 BGE-Reranker 或 Qwen-Reranker）进行二次打分排序，把最相关的排在前面，解决向量相似度"语义漂移"的问题。
- **提升召回率 (Multi-Query)**：利用 LLM 将用户的原始问题改写成多个不同表达的查询 (Multi-Query)，分别去检索，然后取并集。这样能覆盖更多潜在的相关文档，避免因提问方式不同导致漏找。
- **智能 Query 改写**：针对多轮对话中的"上下文依赖"、"指代模糊"、"多意图"等复杂情况，先用 LLM 改写成独立完整的 Query 再检索。同时还能识别是否需要"联网搜索"，实现本地知识库与互联网搜索的混合检索路由。
- **知识库工程化**：
    - **问题生成优化**：给知识切片自动生成相关问题 (QA Pairs)，检索时匹配"问题-问题"的相似度，比"问题-原文"更准。
    - **知识沉淀**：从历史对话中自动提取并沉淀新知识。
    - **健康度检查**：用 LLM 自动检查知识库的完整性、时效性和冲突。
    - **版本管理**：像代码一样管理知识库版本，对比差异并评估检索性能。

### 理解
- **Rerank 重排序 (`1.rerank`)**：
  - `1.beg-reranker-使用本地模型.py`：使用 BGE-Reranker 模型对检索结果进行精细化排序，计算 (Query, Doc) 匹配分。
  - `2.qwen3-reranker-使用本地模型.py`：尝试了生成式 Rerank 模型 (Qwen3-Reranker)，通过判断 "yes/no" 的概率来衡量相关性，效果通常更好但开销稍大。
- **召回率提升 (`2.recall-rate`)**：
  - `1.提高召回率.py`：构建了一个使用 `MultiQueryRetriever` 的 RAG 系统，展示了如何通过生成多个相似 Query 来扩大检索范围。
  - `2.测试相似语义改写后的Query.py`：专门测试 Query 改写的效果，看 LLM 能把一个问题变出多少种花样。
- **Query 改写与意图识别 (`3.query-rewriting`)**：
  - `1.多轮对话中五种Query改写.py`：核心脚本。定义了 5 种改写策略：上下文补全、对比明确化、指代消解、多意图拆解、反问转陈述。
  - `2.支持多轮对话Query改写和多Query检索.py`：将上面的改写逻辑集成到 RAG 流程中，实现支持多轮对话的智能问答。
  - `3.Query联网搜索识别与改写.py`：判断用户问题是否涉及"最新"、"价格"等动态信息，如果是则生成联网搜索的关键词。
  - `4.多轮对话Query联网搜索识别与改写以及多Query检索.py`：集大成者，实现了一个"混合检索 RAG 系统"，智能路由本地检索和联网搜索。
- **知识库处理 (`4.knowledge-base-processing`)**：
  - `1.知识库问题生成与检索优化.py`：一种高级优化技巧。为文档切片生成可能的提问，建立"问题索引"，检索时匹配生成的问题，提高准确率。
  - `2.对话知识的沉淀.py`：从客服对话记录中提取事实性知识，过滤闲聊，合并相似条目，自动充实知识库。
  - `3.知识库的健康度检查.py`：利用 LLM 充当"质检员"，找出知识库中缺失、过期或相互冲突的内容，生成健康报告。
  - `4.知识库的版本管理与性能对比.py`：引入工程化思维，对知识库进行版本控制，并量化评估不同版本的检索准确率和响应时间。
